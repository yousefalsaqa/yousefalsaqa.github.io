Document Automation

A Selenium script that turns an 8-hour day of manual document attachment into "press enter and walk away." ~300 documents processed while I do actual work.

Role Sole Developer
Timeline Jan 2026 — Present
Stack Python, Selenium, ChromeDriver
Status In daily use

The Problem

As an Admission Coordinator, part of my job is attaching documents to student records in PeopleSoft. Transcripts, test scores, recommendation letters — they come into a shared folder and need to be linked to the right student application.

The manual process:

  1. Open a document from the shared folder
  2. Find the student ID in the filename
  3. Search for that student in PeopleSoft
  4. Navigate to their application
  5. Upload the document
  6. Repeat ~300 times a day

A full 8-hour day of clicking. Open, search, click, upload, next. Mind-numbing repetition that a computer should be doing.

The Solution

I wrote a Python script using Selenium that automates the entire workflow. Point it at a folder of documents, let it run, come back to find everything attached.

How It Works

  1. Parse filenames: Extract student ID and document type from standardized filenames
  2. Authenticate: Log into PeopleSoft (handles login with user input once at start)
  3. Search and navigate: Find each student's application automatically
  4. Upload: Attach the document to the correct record
  5. Log everything: Track successes, failures, and edge cases
# Core workflow (simplified)
def process_document(filepath):
    student_id, doc_type = parse_filename(filepath)
    
    # Search for student
    driver.get(f"{PEOPLESOFT_URL}/search?q={student_id}")
    wait_for_results()
    
    # Navigate to application
    click_first_result()
    navigate_to_documents_tab()
    
    # Upload
    upload_file(filepath)
    
    # Verify and log
    if verify_upload():
        log_success(student_id, doc_type)
        move_to_completed(filepath)
    else:
        log_failure(student_id, doc_type)
        move_to_review(filepath)

Technical Challenges

PeopleSoft's Dynamic DOM

PeopleSoft loads content dynamically with unpredictable element IDs. I couldn't rely on static selectors. Solution: wait for specific text content and use relative XPath queries based on visible labels.

Rate Limiting

Too fast and PeopleSoft throttles you. Too slow and you're not saving time. I added randomized delays (2-5 seconds between actions) that feel "human" and avoid triggering rate limits.

Error Recovery

Networks fail. Pages timeout. Students don't exist. The script handles all of this gracefully:

  • Automatic retry on network errors (up to 3 times)
  • Failed documents moved to an "errors" folder
  • Detailed CSV log — every failure includes why it failed, so I can fix the root cause instead of just retrying blindly
  • Resume from where it left off if interrupted

Filename Parsing

Document filenames aren't always consistent. I built a parser that handles multiple formats:

  • 12345678_Transcript.pdf
  • Transcript_12345678.pdf
  • 12345678 - Official Transcript (1).pdf

The Lesson: Automation Is a Conversation

Here's what I didn't expect: automation changes your relationship with a process. Before the script, I'd click through documents without thinking. After? I started noticing patterns. Why do these files keep failing? Because the naming convention from this source is inconsistent. Why are these taking longer? Because that PeopleSoft page loads slower than others.

The script forced me to understand the process deeply — every edge case, every failure mode. And that understanding led to improvements beyond just the automation. I started pushing back on inconsistent naming conventions. I flagged slow pages to IT. The automation wasn't just about saving time; it was about exposing everything that was broken in the workflow.

That's the thing about automation: it doesn't hide problems, it amplifies them. Every exception you have to handle is a question about why that exception exists in the first place.

What I Learned

  • Automation exposes broken processes. If your script needs 50 special cases, maybe the process needs fixing, not just automating.
  • Logging is non-negotiable. A CSV of failures with reasons means I spend 10 minutes fixing root causes instead of hours re-running blind.
  • Selenium isn't elegant, but it works. No API access? No problem. If a human can click it, Selenium can click it.
