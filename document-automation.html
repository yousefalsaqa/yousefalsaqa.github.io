Document Automation

A Selenium script that attaches documents to student applications automatically. 478 documents in 21 hours, no manual clicking required.

Role Sole Developer
Timeline Built in 1 day
Stack Python, Selenium
Status Still running

The Problem

In the admission office, we receive documents that need to be attached to student applications in our CRM (Salesforce). Transcripts, recommendation letters, test scores — hundreds of files.

The manual process:

  1. Open a document from the shared folder
  2. Find the student ID in the filename
  3. Search for that student in Salesforce
  4. Navigate to their application
  5. Upload the document
  6. Repeat 500 times

Each document takes 2-3 minutes manually. At peak times, we'd have backlogs of 500+ documents. That's potentially 25 hours of clicking.

The Solution

I wrote a Python script using Selenium that automates the entire workflow. Point it at a folder of documents, let it run, come back to find everything attached.

How It Works

  1. Parse filenames: Extract student ID and document type from standardized filenames
  2. Authenticate: Log into Salesforce (handles MFA with user input once at start)
  3. Search and navigate: Find each student's application automatically
  4. Upload: Attach the document to the correct record
  5. Log everything: Track successes, failures, and edge cases
# Core workflow (simplified)
def process_document(filepath):
    student_id, doc_type = parse_filename(filepath)
    
    # Search for student
    driver.get(f"{SALESFORCE_URL}/search?q={student_id}")
    wait_for_results()
    
    # Navigate to application
    click_first_result()
    navigate_to_documents_tab()
    
    # Upload
    upload_file(filepath)
    
    # Verify and log
    if verify_upload():
        log_success(student_id, doc_type)
        move_to_completed(filepath)
    else:
        log_failure(student_id, doc_type)
        move_to_review(filepath)

Technical Challenges

Salesforce's Dynamic DOM

Salesforce loads content dynamically with unpredictable element IDs. I couldn't rely on static selectors. Solution: wait for specific text content and use relative XPath queries based on visible labels.

Rate Limiting

Too fast and Salesforce throttles you. Too slow and you're not saving time. I added randomized delays (2-5 seconds between actions) that feel "human" and avoid triggering rate limits.

Error Recovery

Networks fail. Pages timeout. Students don't exist. The script handles all of this gracefully:

  • Automatic retry on network errors (up to 3 times)
  • Screenshots on failure for debugging
  • Separate folders for completed, failed, and needs-review documents
  • Resume from where it left off if interrupted

Filename Parsing

Document filenames aren't always consistent. I built a parser that handles multiple formats:

  • 12345678_Transcript.pdf
  • Transcript_12345678.pdf
  • 12345678 - Official Transcript (1).pdf

Impact

478 Documents processed (first run)
21hr Runtime (unattended)
~25hr Manual time saved
97% Success rate

The script runs overnight or on weekends. Documents that would have taken a full workday to process are done without any human intervention.

The 3% failure rate is mostly edge cases — students who withdrew, duplicate submissions, corrupt files. These get flagged for human review, which is exactly what a human should be spending time on.

Still running: I built this as a Recruitment Rep. Now as an Admission Coordinator, I'm still using it — and extending it to handle new document types.

What I Learned

  • Automation doesn't have to be perfect. 97% automated with good logging beats 0% automated while waiting for a "proper" solution.
  • Build for maintenance. Clear logging and error handling mean I can debug issues months later without remembering every detail.
  • Sometimes the boring solution is the right one. Selenium isn't elegant, but it works with any web interface without API access.
