Energy Intensity Index Predictor

A machine learning model that predicts EII for a chemical refinery's 4 plants — matching manual calculations that took 2.5 weeks, with 99.3% accuracy.

Role Sole Developer
Timeline Jan 2024 — Apr 2024
Stack Python, PyTorch, sklearn
Context Freelance project

The Problem

A chemical refinery with 4 plants needed to calculate their Energy Intensity Index (EII) regularly — it's a key metric for regulatory compliance and internal efficiency tracking. The problem? The manual calculation process took about 2.5 weeks every time.

They had years of historical data and all the input parameters they normally used for EII calculations. The question was whether machine learning could speed up the process while maintaining accuracy.

The Solution

I built a model that takes the same inputs they'd use for manual EII calculations — production volumes, energy consumption by type, equipment utilization, ambient conditions — plus historical data from previous years. The model learns the relationship between these inputs and the final EII values.

Approach: Try Everything, Keep What Works

I tried both neural networks and random forests. Neural networks are great at capturing complex non-linear relationships. Random forests are more robust to outliers and easier to interpret. I ended up using both and combining their predictions — each catches things the other misses.

class EIINetwork(nn.Module):
    def __init__(self, input_dim):
        super().__init__()
        self.network = nn.Sequential(
            nn.Linear(input_dim, 128),
            nn.ReLU(),
            nn.Dropout(0.3),
            nn.Linear(128, 64),
            nn.ReLU(),
            nn.Dropout(0.2),
            nn.Linear(64, 32),
            nn.ReLU(),
            nn.Linear(32, 1)
        )
    
    def forward(self, x):
        return self.network(x)

# Train 10 networks with different seeds
models = []
for seed in range(10):
    torch.manual_seed(seed)
    model = EIINetwork(X.shape[1])
    train(model, X_train, y_train)
    models.append(model)

# Ensemble prediction
def ensemble_predict(X):
    preds = [model(X) for model in models]
    return torch.stack(preds).mean(dim=0)

Stage 2: Random Forest

A random forest model trained on the same data provides a second opinion. Tree-based methods handle outliers and non-linear interactions differently than neural networks, providing complementary predictions.

Final predictions combine both models using weights determined by validation MSE:

# Weight by inverse MSE (better models get more weight)

Data Processing

Feature Engineering

Raw sensor data needed significant preprocessing:

  • Rolling statistics: 1-hour, 4-hour, and 24-hour rolling means and standard deviations capture temporal patterns
  • Lag features: Previous time steps' values help the model understand trends
  • Cyclical encoding: Hour of day and day of week encoded as sine/cosine pairs
  • Interaction terms: Products of related features (e.g., temperature × humidity)

MAD Normalization

Standard z-score normalization is sensitive to outliers. I used Median Absolute Deviation (MAD) normalization instead:

def mad_normalize(X):
    median = np.median(X, axis=0)
    mad = np.median(np.abs(X - median), axis=0)
    # Avoid division by zero
    mad[mad == 0] = 1
    return (X - median) / (1.4826 * mad)

The 1.4826 factor makes MAD consistent with standard deviation for normal distributions, while remaining robust to outliers.

Validation Strategy

Energy data has temporal structure — random train/test splits would leak future information. I used:

Time-based Split

Training on earlier data, testing on later data

5-Fold CV

Rolling window cross-validation respecting time order

Gap Period

1-week gap between train and test prevents leakage

Multiple Metrics

MSE, MAE, and R² reported for complete picture

The Struggle: ML Is Educated Guessing

Here's what nobody tells you about machine learning: it's a lot of informed guessing. Change the learning rate? Maybe it gets better, maybe worse. Add a layer? Flip a coin. Different random seed? Completely different results.

I spent hours tweaking hyperparameters, trying different architectures, adjusting regularization. Most changes made no difference. Some made things worse. A few made things better, and I couldn't always explain why.

Eventually I learned to be systematic about it — grid search over parameters, track every experiment, trust the validation metrics over intuition. But there's still an element of trial and error that feels unscientific. You just have to make peace with it.

What I Learned

  • Track everything. When you're trying 50 different configurations, you need logs. What parameters, what results, what changed. Otherwise you're wandering in the dark.
  • Data quality > model complexity. Better feature engineering improved results more than fancy architectures. Clean data in, useful predictions out.
  • Combine approaches. Neural networks and random forests make different kinds of mistakes. Using both and averaging catches errors neither would catch alone.
