Admission Coordinator Assistant

A Tkinter and Selenium tool that takes the entire admission decision workflow off the coordinator's hands except for the one part that still needs a human: reading the documents and calculating the average. Everything else gets handled automatically and the tool moves to the next student.

Role Sole Developer
Timeline Jan 2026 to Present
Stack Python, Tkinter, Selenium
Status In daily use

The Problem

Reviewing an admission file isn't one job, it's about a dozen. The coordinator has to open a folder of documents, calculate an average from the transcripts, enter that average into PeopleSoft, sign the document off, possibly change the curriculum if the student is being moved between programs, possibly add or remove a document requirement on their record, then click through to the next student in the queue.

The actual judgment, looking at the transcripts and figuring out the average, is maybe 10% of the work. The other 90% is mechanical clicking that a computer should be doing. Multiply by hundreds of students.

The Solution

I built a tool that handles everything except the part that requires actual judgment.

The New Workflow

  1. Tool loads the next student in the queue and opens their documents
  2. Coordinator reads the docs and calculates the average
  3. Coordinator types the average into the tool
  4. Tool does the rest: enters the data into PeopleSoft, signs the documents off, applies any curriculum changes, updates document requirements, and advances to the next student

The coordinator's day goes from "open, search, type, click, click, sign, click, change curriculum, save, next" to "read, calculate, type number, repeat." The boring stuff is gone.

What It Handles

Document Opening

Pulls the right documents for the next student in the queue and opens them in order so the coordinator can read straight through

Data Entry

Takes the calculated average and pushes it into the right PeopleSoft fields for the right application

Document Sign-Off

Marks documents as reviewed and signed off without manual clicking through each one

Curriculum Changes

Handles program transfers and curriculum updates as part of the same flow

Document Requirements

Adds, removes, or modifies document requirements on a student's record without leaving the tool

Queue Management

Auto-advances to the next student so there's no context switching back to PeopleSoft to find them

How It's Built

Tkinter Front End

The coordinator-facing piece is a small Tkinter window: which student is up, what documents are loaded, where to type the average, what mode the tool is in (standard review, curriculum change, doc requirement update). Tkinter is unglamorous, but it ships, runs on whatever laptop the coordinator has, and doesn't need IT to approve a deployment.

Selenium Back End

Behind the scenes, Selenium drives PeopleSoft the way a coordinator would, with the same logged-in browser session. Each action (data entry, sign-off, curriculum change, requirement edit) is its own routine with its own retry and logging path.

State Per Student

Every student in the queue is a small state object: which docs are open, what's been entered, what's been signed off, what curriculum or requirement changes are pending. If anything fails partway through, the tool knows where it stopped and either retries or kicks the student into a manual-review pile so nothing falls silently through the cracks.

Built On Top of the Document Automation Script

The original document-attachment script (the Selenium tool that just attached documents to records) was the seed. Once that proved the PeopleSoft side was automatable, this tool extended the same pattern across the rest of the workflow: reading queues, entering data, signing things off, moving things between programs.

The attachment script is still in daily use. The Coordinator Assistant is what came next, the same idea applied to everything else the role spends time on.

What's Next

The tool is currently used by me, in my own role. The natural next step is rolling it out to other coordinators and using it as a training aid for new hires, since it enforces the workflow and removes the parts where new coordinators tend to make data-entry mistakes. That's a conversation for the coming months.

What I Learned

  • Automate around the human, not the human. The coordinator's judgment is the valuable part. Build tools that hand off everything except that.
  • Selenium is glue, not a strategy. No API access? Selenium works. But every Selenium step is a future maintenance bill, so log everything and design for graceful failure.
  • Tools change roles. Once the mechanical work is gone, the role becomes "make decisions and keep the queue moving." That's a different (and better) job than it started as.
  • Build it for one user first. Don't generalize until you've used it in your own role for months. Generalizing too early bakes in assumptions that don't survive contact with a second user.
