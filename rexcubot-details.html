Rexcubot Competition Robot

A competition robot for Tronassic Park featuring a parallel linkage scoop, precision gripper, and hybrid autonomous/manual control. Designed to retrieve objects and rescue "Ken."

Role Mechanical & Software Lead
Timeline ELEC 303 Project
Team Team Rexcubot (4)
Result Competition finalist

The Challenge

Tronassic Park is a robotics competition with multiple objectives:

  • Collect dinosaur eggs: Small foam balls scattered around the arena
  • Rescue Ken: A doll stuck in a "tar pit" that needs to be lifted out
  • Return to base: Navigate back with all collected items
  • Time limit: 3 minutes to complete all objectives

The challenge required both broad collection (scooping multiple eggs at once) and precision manipulation (gently lifting Ken without knocking him over).

Mechanical Design

Parallel Linkage Scoop

The front scoop uses a 4-bar parallel linkage. Why parallel? It keeps the scoop floor level throughout its range of motion — eggs don't roll out when lifting.

Design considerations:

  • Link lengths: Optimized for maximum lift height while keeping the mechanism compact
  • Material: 6061 aluminum for strength-to-weight ratio
  • Actuation: Single servo with 20kg-cm torque, mounted at the base for low center of gravity
  • Edge design: Thin leading edge slides under objects; raised sides prevent spilling

Precision Gripper

Ken required a different approach. The gripper uses a rack-and-pinion mechanism for linear opening/closing motion with precise position control.

The gripper fingers have foam padding to:

  • Increase friction for secure grip
  • Conform to Ken's irregular shape
  • Prevent damage during manipulation

Control System

The robot operates in two modes: autonomous line-following and manual gamepad control.

Autonomous Mode

IR sensors on the underside detect the white arena lines against the dark floor. A PID controller keeps the robot centered on the path.

// Line following PID controller
float line_follow_update(float error) {
    static float integral = 0;
    static float prev_error = 0;
    
    integral += error * dt;
    integral = constrain(integral, -MAX_INTEGRAL, MAX_INTEGRAL);
    
    float derivative = (error - prev_error) / dt;
    prev_error = error;
    
    return Kp * error + Ki * integral + Kd * derivative;
}

// Sensor reading to error conversion
float get_line_error() {
    // 5 IR sensors: far_left, left, center, right, far_right
    int weighted_sum = -2*s[0] + -1*s[1] + 0*s[2] + 1*s[3] + 2*s[4];
    int sum = s[0] + s[1] + s[2] + s[3] + s[4];
    
    if (sum == 0) return last_error;  // Line lost, keep turning
    return (float)weighted_sum / sum;
}

Software Architecture

The robot's firmware uses a state machine architecture for predictable behavior:

enum State {
    IDLE,
    LINE_FOLLOW,
    MANUAL_CONTROL,
    SCOOP_DEPLOY,
    GRIPPER_OPERATE,
    RETURN_HOME
};

void loop() {
    // Read all sensors
    update_sensors();
    
    // Process state transitions
    switch (current_state) {
        case IDLE:
            if (start_button_pressed()) {
                transition_to(LINE_FOLLOW);
            }
            break;
            
        case LINE_FOLLOW:
            if (manual_override_received()) {
                transition_to(MANUAL_CONTROL);
            }
            if (object_detected()) {
                transition_to(SCOOP_DEPLOY);
            }
            execute_line_follow();
            break;
        
        // ... other states
    }
    
    // Update actuators
    update_motors();
}

What I Learned

  • Mechanism choice is fundamental. The parallel linkage made everything else easier. A bad mechanism can't be fixed with good software.
  • Test under competition conditions. Our robot worked perfectly in the lab but struggled with different lighting at competition. More environmental testing was needed.
  • Hybrid control is powerful. Autonomous for the boring parts (line following), manual for the hard parts (precision manipulation). Best of both worlds.
