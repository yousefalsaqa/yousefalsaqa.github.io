Rexcubot Competition Robot

A Jurassic Park-themed competition robot: autonomous navigation, dinosaur collection, and a Ken rescue mission. Also a lesson in picking up the slack when teammates don't deliver.

Role Software & Integration Lead
Timeline Jan 2024 — May 2024
Course Third Year Design
Team 3 people (sort of)

The Challenge

Jurassic Park-themed robotics competition on a multi-level stage. Three distinct phases:

  • Level 1 (Autonomous): Navigate to find and pass through the door to level 2
  • Level 2 (Manual): Use a bucket to shove toy dinosaurs and deposit them into a cage
  • Level 3 (Manual): Go up an incline, rescue Ken from a platform, and return to the starting position with Ken

My responsibility: all the software — the autonomous navigation for level 1 and the controller integration for manual control in levels 2 and 3.

Mechanical Design

Parallel Linkage Scoop

The front scoop uses a 4-bar parallel linkage. Why parallel? It keeps the scoop floor level throughout its range of motion — eggs don't roll out when lifting.

Design considerations:

  • Link lengths: Optimized for maximum lift height while keeping the mechanism compact
  • Material: 6061 aluminum for strength-to-weight ratio
  • Actuation: Single servo with 20kg-cm torque, mounted at the base for low center of gravity
  • Edge design: Thin leading edge slides under objects; raised sides prevent spilling

Precision Gripper

Ken required a different approach. The gripper uses a rack-and-pinion mechanism for linear opening/closing motion with precise position control.

The gripper fingers have foam padding to:

  • Increase friction for secure grip
  • Conform to Ken's irregular shape
  • Prevent damage during manipulation

Control System

The robot operates in two modes: autonomous line-following and manual gamepad control.

Autonomous Mode

IR sensors on the underside detect the white arena lines against the dark floor. A PID controller keeps the robot centered on the path.

// Line following PID controller
float line_follow_update(float error) {
    static float integral = 0;
    static float prev_error = 0;
    
    integral += error * dt;
    integral = constrain(integral, -MAX_INTEGRAL, MAX_INTEGRAL);
    
    float derivative = (error - prev_error) / dt;
    prev_error = error;
    
    return Kp * error + Ki * integral + Kd * derivative;
}

// Sensor reading to error conversion
float get_line_error() {
    // 5 IR sensors: far_left, left, center, right, far_right
    int weighted_sum = -2*s[0] + -1*s[1] + 0*s[2] + 1*s[3] + 2*s[4];
    int sum = s[0] + s[1] + s[2] + s[3] + s[4];
    
    if (sum == 0) return last_error;  // Line lost, keep turning
    return (float)weighted_sum / sum;
}

Software Architecture

The robot's firmware uses a state machine architecture for predictable behavior:

enum State {
    IDLE,
    LINE_FOLLOW,
    MANUAL_CONTROL,
    SCOOP_DEPLOY,
    GRIPPER_OPERATE,
    RETURN_HOME
};

void loop() {
    // Read all sensors
    update_sensors();
    
    // Process state transitions
    switch (current_state) {
        case IDLE:
            if (start_button_pressed()) {
                transition_to(LINE_FOLLOW);
            }
            break;
            
        case LINE_FOLLOW:
            if (manual_override_received()) {
                transition_to(MANUAL_CONTROL);
            }
            if (object_detected()) {
                transition_to(SCOOP_DEPLOY);
            }
            execute_line_follow();
            break;
        
        // ... other states
    }
    
    // Update actuators
    update_motors();
}

Results

~9 sec Autonomous level time
3 Competition levels

The good: The autonomous portion — my part — was elite. About 9 seconds to navigate level 1 and find the door. Fast, consistent, worked every time.

The bad: The bucket design had a fatal flaw. The teammate who designed it didn't account for the lip at the bottom of the cage or how the weight of the dinosaurs would push down on the bucket. When we tried to dump the dinos into the cage, the bucket couldn't get over the lip. That cost us the dino collection phase entirely.

The Team Dynamics Problem

We were technically a team of 3. In practice, it was closer to 2.

The third team member barely attended meetings. His one major task — designing the gripper for the Ken rescue — was assigned in week 6. He took the entire semester. When he finally delivered, the design didn't make mechanical sense. I ended up designing a completely different approach in the final weeks.

It's frustrating. You want to trust your teammates, give them space to own their work. But at some point, you have to recognize that the project is going to fail if you don't step in. That's not a comfortable position, but it's reality.

What I Learned

  • Leadership sometimes means picking up the slack. Not because you want to, but because the project needs it. Document everything so it's clear who did what.
  • Mechanical design matters more than software. My autonomous code was perfect, but it couldn't fix a bucket that couldn't clear a lip. Integration means nothing if the hardware doesn't work.
  • Hybrid control is powerful. Autonomous for repeatable tasks, manual for the parts that need human judgment. Best of both worlds.
